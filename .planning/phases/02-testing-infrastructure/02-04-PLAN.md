  <action>
Create comprehensive unit tests for executeApiAudio function.

**Testing strategy:** Mock getActiveApiClient via vi.mock to return a mock REWClient object. Do NOT use MSW - these are unit tests that mock at the function level.

**Test structure:**
```typescript
import { describe, it, expect, vi, beforeEach } from 'vitest';
import { executeApiAudio } from './api-audio.js';
import { REWApiError } from '../api/rew-api-error.js';

// Mock getActiveApiClient
vi.mock('./api-connect.js', () => ({
  getActiveApiClient: vi.fn()
}));
import { getActiveApiClient } from './api-connect.js';
const mockGetActiveApiClient = vi.mocked(getActiveApiClient);
```

**Test cases to cover:**

1. **No connection tests:**
   - Returns connection_error when getActiveApiClient returns null

2. **status action tests:**
   - Returns audio status with all fields populated
   - Handles null/undefined values from API gracefully

3. **list_devices action tests:**
   - Returns input devices, output devices, sample rates arrays
   - Handles empty device lists

4. **set_input action tests:**
   - Returns validation error when device not provided
   - Returns success when setJavaInputDevice succeeds
   - Returns success with failure message when setJavaInputDevice returns false

5. **set_output action tests:**
   - Same pattern as set_input

6. **set_sample_rate action tests:**
   - Returns validation error when sample_rate not provided
   - Returns success when setSampleRate succeeds
   - Returns success with failure message when setSampleRate returns false

7. **Error handling tests:**
   - Returns validation_error for ZodError (invalid action)
   - Returns typed error for REWApiError with correct suggestion mapping
   - Returns internal_error for unknown errors
