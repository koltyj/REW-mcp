  <name>Task 1: Create optimization tool with get_recommendation action</name>
  <files>src/tools/optimize-room.ts</files>
  <action>
Create `src/tools/optimize-room.ts` with:

1. **Input Schema (Zod)**:
   - `action: z.enum(['get_recommendation', 'validate_adjustment', 'check_progress'])`
   - `measurement_id: z.string()` - Current/post measurement
   - `pre_measurement_id: z.string().optional()` - For validation action
   - `session_id: z.string().optional()` - For session-based tracking
   - `left_measurement_id: z.string().optional()` - For L/R analysis
   - `right_measurement_id: z.string().optional()` - For L/R analysis
   - `sub_measurement_id: z.string().optional()` - For sub analysis
   - `room_dimensions: z.object({ length, width, height }).optional()`
   - `target_frequency_hz: z.number().optional()` - For validation, which issue to check
   - `target_category: z.string().optional()` - For validation, issue category

2. **Output Types**:
   - GetRecommendationResult: Single prioritized recommendation with next_steps guidance
   - ValidateAdjustmentResult: Improvement classification with numbers and next action
   - CheckProgressResult: Zone-based success criteria evaluation

3. **get_recommendation action**:
   - Get measurement from store
   - Run analyze-room style detection (peaks, nulls, optional L/R, sub)
   - Prioritize issues using existing prioritizeIssues()
   - Select top issue (biggest remaining problem)
   - Generate recommendation using generatePlacementRecommendation/generateSubRecommendation
   - Return single recommendation with:
     - recommendation: PlacementRecommendation
     - priority_rank: 1 (always - we give one at a time)
     - total_issues: number of issues detected
     - next_steps: "Make this adjustment, then re-measure and call validate_adjustment"

Important: One recommendation at a time per CONTEXT.md ("suggest, measure, evaluate, then next")

Import from:
- `../optimization/recommendations.js`
- `../optimization/validation.js`
- `../optimization/success-criteria.js`
- `../store/measurement.js`
- `../analysis/peaks-nulls.js`
- `../interpretation/prioritization.js`
  </action>
  <verify>npx tsc --noEmit passes</verify>
  <done>get_recommendation action returns single prioritized recommendation</done>

<task type="auto">
  <name>Task 2: Add validate_adjustment and check_progress actions</name>
  <files>src/tools/optimize-room.ts</files>
  <action>
Extend `src/tools/optimize-room.ts` with:

1. **validate_adjustment action**:
   - Requires: measurement_id (post), pre_measurement_id (before adjustment)
   - Optional: target_frequency_hz, target_category (to focus validation)
   - Get both measurements from store
   - Call validateAdjustment() from validation module
   - Return ValidationResult with:
     - improvement_type: 'success' | 'partial' | 'unchanged' | 'worsened'
     - summary: Plain language with dB and percentage
     - next_action: What to do next
     - If success/partial: "Good progress. Call get_recommendation for next issue."
     - If worsened: "Try moving the opposite direction, then re-measure."
     - If unchanged: "Try a different adjustment or call get_recommendation for alternative."

2. **check_progress action**:
   - Requires: measurement_id
   - Optional: left_measurement_id, right_measurement_id, sub_measurement_id
   - Call evaluateSuccessCriteria() from success-criteria module
   - Return SuccessCriteriaResult with:
     - smoothness, lr_balance, sub_integration zone evaluations
     - overall_zone: worst of all
     - should_stop: boolean
     - progress_summary: Plain language
     - If should_stop: "Target achieved. You've reached +-X dB - further gains will be marginal."
     - If not: "Continue optimization. Focus on [worst zone] next."

3. **Error handling**:
   - measurement_not_found: Return helpful error with suggestion to use get_status
   - validation_error: Return Zod errors formatted for user
   - For validate_adjustment without pre_measurement_id: Return error suggesting required parameter

4. **Response structure**:
   - status: 'success' | 'error'
   - data: action-specific result type
   - error_type, message for errors
  </action>
  <verify>npx tsc --noEmit passes</verify>
  <done>All three actions work with proper error handling</done>
</task>
