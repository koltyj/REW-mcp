  <name>Task 1: Create validation module</name>
  <files>src/optimization/validation.ts</files>
  <action>
Create `src/optimization/validation.ts` with:

1. **ValidationResult interface**:
   - `improvement_type: 'success' | 'partial' | 'unchanged' | 'worsened'`
   - `metric_name: string` - What was measured ("Bass variance", "Peak at 125Hz")
   - `pre_value_db: number` - Before adjustment
   - `post_value_db: number` - After adjustment
   - `improvement_db: number` - Absolute change
   - `improvement_percent: number` - Proportional change
   - `summary: string` - Plain language ("Variance improved from 8.2dB to 4.1dB (50% reduction)")
   - `next_action: string` - What to do next ("Proceed to next issue" or "Try moving opposite direction")
   - `explanation: string` - Physics context for why it worked/didn't work

2. **validateAdjustment(preMeasurement: StoredMeasurement, postMeasurement: StoredMeasurement, targetIssue: { frequency_hz: number; category: string }): ValidationResult**
   - Calculate metric based on issue category:
     - For peaks: Compare peak deviation at target frequency (5 Hz tolerance)
     - For nulls: Compare null depth at target frequency
     - For sub_integration: Compare crossover region variance (40-100 Hz)
     - For lr_symmetry: Compare L/R deviation
     - Default: Compare overall 40-200 Hz variance
   - Use proportional threshold (50%+ = success) from RESEARCH.md
   - Use context-dependent unchanged threshold (reuse getUnchangedThreshold logic):
     - Small issues (<6 dB): <1 dB = unchanged
     - Medium issues (6-10 dB): <2 dB = unchanged
     - Large issues (>10 dB): <3 dB = unchanged
   - Classify worsened: improvement_percent < -10 (per RESEARCH.md)
   - Generate plain language summary with both dB and percentage
   - If worsened: next_action = "This worsened the response. Try moving the opposite direction."
   - If success: next_action = "Good improvement. Ready to address next issue."
   - If partial: next_action = "Some improvement. Consider fine-tuning or moving to next issue."
   - If unchanged: next_action = "No significant change detected. Try a different adjustment or element."

3. **Helper: calculateMetricForCategory(measurement: StoredMeasurement, category: string, frequencyHz: number): number**
   - Extract relevant metric based on category
   - Use detectPeaks/detectNulls for peak/null categories
   - Use quick_stats.variance_20_200hz_db for variance
   - Return the specific deviation or variance value

4. **Helper: getUnchangedThreshold(deviation_db: number): number**
   - Copy from glm-comparison.ts (or import if exported)
   - <6 dB: return 1
   - <10 dB: return 2
   - >= 10 dB: return 3

Import:
- StoredMeasurement from `../types/index.js`
- detectPeaks, detectNulls from `../analysis/peaks-nulls.js`

<task type="auto">
  <name>Task 2: Create success criteria module</name>
  <files>src/optimization/success-criteria.ts</files>
  <action>
Create `src/optimization/success-criteria.ts` with:

1. **SuccessZone type**: `'good' | 'acceptable' | 'needs_work'`

2. **SuccessCriteriaResult interface**:
   - `smoothness: { zone: SuccessZone; variance_db: number; target_db: number; message: string }`
   - `lr_balance: { zone: SuccessZone; max_deviation_db: number; target_db: number; message: string }`
   - `sub_integration: { zone: SuccessZone; crossover_variance_db: number; target_db: number; message: string }`
   - `overall_zone: SuccessZone` - Worst of the three
   - `should_stop: boolean` - True if all zones are 'good'
   - `progress_summary: string` - Plain language progress ("Smoothness: Good. L/R Balance: Acceptable. Sub: Needs work.")
   - `limitation_note?: string` - If target physically impossible, explain

3. **evaluateSuccessCriteria(measurement: StoredMeasurement, options?: { leftMeasurement?: StoredMeasurement; rightMeasurement?: StoredMeasurement; subMeasurement?: StoredMeasurement }): SuccessCriteriaResult**

   Zone thresholds (per CONTEXT.md and RESEARCH.md):
   - **Smoothness** (40-200 Hz variance):
     - good: <= 3 dB (±3 dB target)
     - acceptable: <= 5 dB (±4-5 dB)
     - needs_work: > 5 dB
   - **L/R Balance** (max deviation):
     - good: <= 1 dB (excellent)
     - acceptable: <= 2 dB (good to fair)
     - needs_work: > 2 dB
   - **Sub Integration** (40-100 Hz variance):
     - good: <= 4 dB
     - acceptable: <= 6 dB
     - needs_work: > 6 dB

   Generate messages:
   - good: "Within target (+-X dB) - further gains will be marginal"
   - acceptable: "Close to target (X dB vs Y dB goal) - improvement possible"
   - needs_work: "X dB variance - target is +-Y dB"

   should_stop logic:
   - True if smoothness.zone === 'good' (primary metric)
   - Include note if sub or L/R still needs work but smoothness achieved

   limitation_note:
   - If variance hasn't improved after analysis, note: "Target +-3dB may not be achievable without treatment at [problem frequency]"
   - This is informational, not blocking

4. **Helper: calculateBandVariance(frequencyResponse: FrequencyResponse, minHz: number, maxHz: number): number**
   - Extract SPL values in band
   - Return max - min (range as variance metric)

Update `src/optimization/index.ts`:
